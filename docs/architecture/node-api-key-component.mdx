The `NodeApiKeySection` is a React component in `remote-frontend/src/components/swarm/` that surfaces API key management on the Hive Nodes page. This document covers its architecture, data flow, testing strategy, and integration patterns for developers.

### Component Structure

```
NodeApiKeySection
├── Card > CardHeader              # Title + "Generate API Key" button
├── CardContent
│   ├── Alert (error, conditional)
│   ├── Loader2 (loading, conditional)
│   ├── empty-state text
│   └── ApiKeyItem[]               # One per key
└── Dialog                         # Create flow
    ├── Name input + Create/Cancel
    └── Secret reveal + Copy + show/hide
```

`ApiKeyItem` is an internal subcomponent (not exported) that renders a single key row with name, prefix, status badge, timestamps, and action button.

### Data Flow

The component uses TanStack Query hooks directly (no extracted custom hook):

```typescript
// List query — scoped by organizationId
useQuery({
  queryKey: ['nodeApiKeys', organizationId],
  queryFn: () => nodesApi.listApiKeys(organizationId),
  enabled: !!organizationId,
  staleTime: 30_000,
})

// Mutations — each invalidates the list query on success
useMutation({ mutationFn: nodesApi.createApiKey, ... })
useMutation({ mutationFn: nodesApi.revokeApiKey, ... })
useMutation({ mutationFn: nodesApi.unblockApiKey, ... })
```

All three mutations invalidate `['nodeApiKeys', organizationId]` on success, triggering a refetch of the list.

### State Management

| State | Type | Purpose |
|-------|------|---------|
| `showCreateDialog` | `boolean` | Controls dialog visibility |
| `newKeyName` | `string` | Name input value |
| `createdSecret` | `string \| null` | One-time secret after creation |
| `showSecret` | `boolean` | Toggle secret visibility |
| `copied` | `boolean` | Copy feedback (auto-resets after 2s) |
| `error` | `string \| null` | Mutation error message |
| `pendingKeyIds` | `Set<string>` | Tracks per-key in-flight mutations |

The `pendingKeyIds` Set ensures only the targeted key's button is disabled during a mutation, not all keys.

### Clipboard Handling

The copy flow uses a two-tier strategy:

1. **Primary**: `navigator.clipboard.writeText()` (modern browsers, secure contexts)
2. **Fallback**: `document.execCommand('copy')` via a temporary textarea (HTTP contexts, older browsers)

The textarea fallback is wrapped in `try/finally` to ensure cleanup. The `copyTimeoutRef` tracks the 2-second "Copied!" feedback timer and is cleared on unmount and dialog close.

### Dialog Behavior

The create dialog has two states:

1. **Name input view**: Shows name field + Create/Cancel buttons
2. **Secret reveal view**: Shows the one-time secret + Copy/Reveal/Hide/Done buttons

When `createdSecret` is set, the dialog becomes `uncloseable` (via the Dialog component's prop) — the user must click "Done" to dismiss. This prevents accidental secret loss via Escape or backdrop click.

## API Integration

The component uses the existing `nodesApi` client from `remote-frontend/src/lib/api/nodes.ts`:

| Function | Endpoint | Used By |
|----------|----------|---------|
| `listApiKeys(orgId)` | `GET /v1/nodes/api-keys?organization_id=...` | `useQuery` |
| `createApiKey({ organization_id, name })` | `POST /v1/nodes/api-keys` | `createMutation` |
| `revokeApiKey(keyId)` | `DELETE /v1/nodes/api-keys/:id` | `revokeMutation` |
| `unblockApiKey(keyId)` | `POST /v1/nodes/api-keys/:id/unblock` | `unblockMutation` |

Types are defined in `remote-frontend/src/types/nodes.ts` (lines 46-75): `NodeApiKey`, `CreateNodeApiKeyRequest`, `CreateNodeApiKeyResponse`.

## i18n

All user-facing strings use `useTranslation(['settings', 'common'])` with keys under `settings.swarm.apiKeys.*`. English fallbacks are passed as the second argument to `t()`:

```typescript
t('settings.swarm.apiKeys.title', 'Node API Keys')
```

The `remote-frontend` does not currently initialize an i18n instance — the inline English fallbacks render at runtime. The locale files in `frontend/src/i18n/locales/` are prepared for future localization.

## Composition in Nodes.tsx

The component is composed into `Nodes.tsx` at line 31:

```tsx
{orgId && <NodeApiKeySection organizationId={orgId} />}
```

This renders above the node grid, gated on `orgId` being present. The `useOrganizations` hook in `Nodes.tsx` owns the org-loading logic; the section receives `organizationId` as a prop.

## Design Decisions

| Decision | Rationale |
|----------|-----------|
| No `isAdmin` prop | Hive UI has no role model; all org members see all controls |
| `confirm()` for destructive actions | Matches frontend reference pattern; AlertDialog available later |
| `useQuery`/`useMutation` inline | Component is the only consumer; hook extraction is premature |
| No new Settings page | Section lives on the Nodes page per spec |
| Per-key pending tracking | Prevents disabling all buttons when one key is being acted on |
| `uncloseable` dialog for secret | Prevents accidental secret loss via Escape/backdrop |
