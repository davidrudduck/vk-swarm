# Database Architecture Overview

Vibe Kanban uses a distributed database architecture with two database systems working together:

1. **SQLite** - Local database on each node (developer machine)
2. **PostgreSQL** - Central database on the hive server (shared infrastructure)

## Database Purposes

### SQLite (Local Node)

| Aspect | Description |
|--------|-------------|
| **Purpose** | Local task execution, project management, execution history |
| **Location** | `dev_assets/db.sqlite` (dev) or platform data directory (prod) |
| **Scope** | Single machine |
| **Availability** | Works offline, no network required |
| **Primary Use** | Storing execution logs, git worktree state, local project config |

The local SQLite database is the source of truth for:
- Execution processes and their logs
- Git worktree and branch management
- Task attempt history
- Merge and PR tracking
- Local project configuration

### PostgreSQL (Central Hive)

| Aspect | Description |
|--------|-------------|
| **Purpose** | Organization management, shared task registry, node coordination |
| **Location** | Remote server (e.g., `postgres://hive.example.com/vibe_remote`) |
| **Scope** | Organization-wide, multi-tenant |
| **Availability** | Requires network connection to hive |
| **Primary Use** | Task assignment, user auth, activity feed, node discovery |

The PostgreSQL hive database is the source of truth for:
- Organizations and user accounts
- Shared task definitions and assignments
- Node registration and capabilities
- Organization-wide activity feed
- Authentication and OAuth tokens

## How They Work Together

### Task Lifecycle Example

```bash
1. USER CREATES TASK (via web UI connected to hive)
   в””в”Җв–ә PostgreSQL: INSERT into shared_tasks

2. HIVE ASSIGNS TASK TO NODE (via WebSocket)
   в””в”Җв–ә PostgreSQL: INSERT into node_task_assignments
   в””в”Җв–ә WebSocket: TaskAssign message to node

3. NODE RECEIVES ASSIGNMENT
   в””в”Җв–ә SQLite: INSERT into tasks (linked via shared_task_id)
   в””в”Җв–ә SQLite: INSERT into task_attempts

4. NODE EXECUTES TASK (coding agent runs)
   в””в”Җв–ә SQLite: INSERT into execution_processes
   в””в”Җв–ә SQLite: INSERT into log_entries (streaming)
   в””в”Җв–ә WebSocket: TaskProgress events to hive
   в””в”Җв–ә WebSocket: TaskOutput logs to hive

5. NODE COMPLETES TASK
   в””в”Җв–ә SQLite: UPDATE task_attempts.status
   в””в”Җв–ә SQLite: INSERT into merges (if PR created)
   в””в”Җв–ә WebSocket: TaskStatus update to hive
   в””в”Җв–ә PostgreSQL: UPDATE shared_tasks.status
```

### Data Flow Summary

| Data Type | Primary Location | Sync Direction | Method |
|-----------|-----------------|----------------|--------|
| Task definitions | PostgreSQL | Bidirectional | WS activity stream + durable outbox (ADR-0007) |
| Task status | Both | Bidirectional | WS activity stream + durable outbox |
| Task assignments | PostgreSQL | Hive вҶ’ Node | WebSocket (lease grant/revoke) |
| Execution logs | SQLite | Node вҶ’ Hive | Durable outbox вҶ’ `OpBatch` |
| Execution history | SQLite | Node only | Not synced |
| Node status | PostgreSQL | Node вҶ’ Hive | WebSocket (heartbeat) |
| Organizations/Users | PostgreSQL | Hive only | Not synced |

## Key Design Decisions

### Why Two Databases?

1. **Offline Capability**: Nodes can work without network connectivity
2. **Performance**: Execution logs stay local, avoiding network latency
3. **Scalability**: Execution data doesn't burden the central server
4. **Privacy**: Sensitive code and logs can stay on local machines
5. **Resilience**: Node failures don't affect other nodes or the hive

### Why SQLite Locally?

- Zero configuration, embedded database
- Fast local reads/writes for streaming logs
- Single file, easy to backup and restore
- Works on all platforms (macOS, Linux, Windows)

### Why PostgreSQL Remotely?

- Robust multi-user support
- Advanced features (partitioning, logical replication)
- Logical replication support for ElectricSQL (non-task sync)
- Proven reliability for production workloads

## Project Tables

### Table Relationships

| Table | Purpose | Status |
|-------|---------|--------|
| `swarm_projects` | Explicitly shared projects across swarm nodes | **Canonical source of truth** |
| `projects` | Legacy hive-level projects | **Deprecated** |
| `node_local_projects` | Inventory of all node projects | For discovery/linking |

### Migration from projects to swarm_projects

The system is transitioning from the legacy `projects` table to `swarm_projects` as the canonical source:

**Activity Tracking:**
- Activity events now have a `swarm_project_id` column linking to `swarm_projects`
- The `project_id` column is retained for backward compatibility
- API consumers should migrate to using `swarm_project_id` parameter

**Access Control:**
- Swarm project access is verified via organization membership
- Uses `SwarmProjectRepository::organization_id()` for lookups
- Replaces legacy project access checks

**Task Management:**
- Tasks are now linked to swarm projects via `shared_tasks.swarm_project_id`
- Activity counters track swarm project metrics

See the [Activity API documentation](../../api/activity.md) for implementation details and migration guidance.
