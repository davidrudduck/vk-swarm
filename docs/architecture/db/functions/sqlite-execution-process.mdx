# Execution Process Functions

Functions for managing execution processes in the local SQLite database. Execution processes track individual runs of scripts and coding agents within a task attempt.

**Source:** `crates/db/src/models/execution_process.rs`

## Process Recovery Functions

These functions support automatic crash recovery on server startup. They manage the `resume_state` column and the orphan-cleanup sweep.

### mark_orphaned_as_failed

Mark execution processes that were left in `status='running'` by a previous server crash as failed.

**Signature:**
```rust
async fn mark_orphaned_as_failed(
    pool: &SqlitePool,
    server_instance_id: &str,
) -> Result<u64, sqlx::Error>
```

**Parameters:**
| Name | Type | Description |
|------|------|-------------|
| `pool` | `&SqlitePool` | Database connection pool |
| `server_instance_id` | `&str` | Current server instance ID (rows owned by this instance are excluded) |

**Returns:** Count of rows updated.

**Guard clause:** Rows with `resume_state IN ('pending', 'resumed')` are excluded — these are under active crash-recovery and must not be prematurely marked failed.

**Used By:**
- `services/container.rs` - `cleanup_orphan_executions` on startup

---

### set_resume_state

Set the `resume_state` column on an execution process row.

**Signature:**
```rust
async fn set_resume_state(
    pool: &SqlitePool,
    id: Uuid,
    state: &str,
) -> Result<(), sqlx::Error>
```

**Valid values:**

| Value | Set by | Meaning |
|---|---|---|
| `pending` | `cleanup_orphan_executions` | Fence complete; crash-resume is being prepared (or process is in D-state) |
| `resumed` | `resume_execution` | Crash-resume has been dispatched to the executor |
| `abandoned` | `cleanup_orphan_executions` | Recovery was skipped (PID reuse detected, or resume failed) |

`NULL` is the initial state for all newly created rows and means the process has not been through crash-recovery logic.

**Used By:**
- `services/container.rs` - `cleanup_orphan_executions`, `resume_execution`

### increment_fence_attempt_count

Increment the D-state retry counter for an execution process. Called each time
`process_fence::fence()` returns `CouldNotKill` (process survived SIGKILL).

**Signature:**
```rust
async fn increment_fence_attempt_count(
    pool: &SqlitePool,
    id: Uuid,
) -> Result<(), sqlx::Error>
```

**Parameters:**
| Name | Type | Description |
|------|------|-------------|
| `pool` | `&SqlitePool` | Database connection pool |
| `id` | `Uuid` | Process ID |

**Returns:** `Ok(())` on success.

**Notes:**
- Implemented as `SET fence_attempt_count = fence_attempt_count + 1` — atomic increment
  without a read-modify-write cycle.
- The counter persists across server restarts (unlike an in-memory counter), which is required
  because D-state processes are only encountered during crash/restart cycles.

**Used By:**
- `services/container.rs` — `cleanup_orphan_executions`, `CouldNotKill` arm

---

### get_fence_attempt_count

Read the current D-state retry counter for an execution process.

**Signature:**
```rust
async fn get_fence_attempt_count(
    pool: &SqlitePool,
    id: Uuid,
) -> Result<i64, sqlx::Error>
```

**Parameters:**
| Name | Type | Description |
|------|------|-------------|
| `pool` | `&SqlitePool` | Database connection pool |
| `id` | `Uuid` | Process ID |

**Returns:** Current value of `fence_attempt_count` (0 by default on new rows).

**Notes:**
- Called immediately after a successful `increment_fence_attempt_count` to check whether the
  escalation threshold (`FENCE_ESCALATION_THRESHOLD = 5`) has been reached.
- On DB read failure after a successful increment, `cleanup_orphan_executions` logs an error
  and skips the threshold check for that cycle — the next restart will retry with the
  correctly incremented value already in the DB.

**Used By:**
- `services/container.rs` — `cleanup_orphan_executions`, escalation warn check
