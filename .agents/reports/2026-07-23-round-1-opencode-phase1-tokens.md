**Date**: 2026-07-23

## Findings

### F1 [Critical] — `base.css` `a` tag styling neutralized by Tailwind Preflight cascade

`base.css` is loaded via `@import` (line 5 of `index.css`) **before** `@tailwind base` (line 7). Both `base.css:33-36` and Tailwind Preflight target `a` with the same specificity (0,0,1). Since Preflight comes later in the cascade, it wins:

```css
/* base.css:33-36 — applies FIRST, overridden */
a {
  color: var(--primary);
  text-decoration: none;
}

/* Tailwind Preflight — applies SECOND, wins */
a {
  color: inherit;
  text-decoration: inherit;
}
```

**Result**: All `<a>` tags in the remote-frontend render with inherited color/text-decoration, not the Midnight Terminal `var(--primary)` cyan. This defeats the token layer's entire purpose for anchors.

**Remediation**: Move `base.css` element styles into an `@layer base` block so they win over (or merge with) Preflight; or wrap in `@layer base` with `@tailwind base` before the `@import`s; or move `@tailwind base` above the `@import`s (but that violates CSS spec).

**Evidence**: `remote-frontend/src/index.css:5,7` order; `remote-frontend/src/styles/tokens/base.css:33-36` vs Tailwind v3 Preflight `a` rule.

### F2 [Critical] — `code, pre, kbd, samp` font-family overridden by Preflight

Same cascade mechanism as F1. `base.css:41-47` sets:
```css
code, pre, kbd, samp {
  font-family: var(--font-code);
}
```

Tailwind Preflight's rule for the same selectors sets `font-family` to the default mono stack. Since Preflight loads after `base.css`, it wins. Code blocks render with the default system mono font family, not the design system's `'JetBrains Mono', 'Chivo Mono', ui-monospace, monospace` stack.

**Evidence**: `remote-frontend/src/styles/tokens/base.css:41-47`; Preflight generated by `@tailwind base`.

### F3 [Important] — `body { line-height }` overridden by Preflight

`base.css:24` sets `line-height: var(--leading-base)` (1.25rem). Tailwind Preflight sets `body { line-height: inherit; }`, inheriting `1.5` from `html`. Since Preflight loads after base.css, the body line-height is `1.5`, not `1.25rem`. The token layer's compact line-height for body text is silently lost.

**Evidence**: `remote-frontend/src/styles/tokens/base.css:24`; Tailwind Preflight `body` rule.

### F4 [Important] — Token layer defines hex values but components use `hsl()` wrapper

The new token layer defines `--vks-*` brand tokens as **hex** and `--vks-*-hsl` as HSL triplets. Semantic aliases (`--status-done`, `--vks-text-dim`, `--surface-card`, `--surface-raised`) resolve to hex:

```css
--vks-emerald: #00ff88;          /* hex */
--vks-emerald-hsl: 152 100% 50%; /* HSL triplet */
--status-done: var(--vks-emerald); /* → #00ff88 (hex) */
```

`NodeCard.tsx` uses `hsl(var(--status-done))`, `hsl(var(--vks-text-dim))`, `hsl(var(--surface-card))`, `hsl(var(--surface-raised))` — all 4 calls produce **invalid CSS** (`hsl(#00ff88)` is not valid). Colors silently fall back to inherited/transparent.

The design source's own `components.css` avoids this by using `hsl(var(--vks-emerald-hsl))` directly — bypassing the semantic aliases. But no existing component does this.

**Failure scenario**: Open `/nodes` in the remote-frontend. NodeCard online dots render with no green color (no `--status-done` resolved), offline dots render with no dim color, the card surface has no background color. The UI appears broken despite the tokens being "correctly" defined.

**Mitigation**: This is Phase 1. Components haven't been updated to use the new token format. But the tokens are in a format that the existing code can't consume — the PR introduces tokens that don't work with the current codebase, and no component migration is in scope.

**Evidence**: `remote-frontend/src/styles/tokens/colors.css:32,84` (hex); `remote-frontend/src/components/swarm/NodeCard.tsx:35,41,48,53` (hsl() wrappers).

### F5 [Important] — `base.test.ts` asserts wrong body token

`base.test.ts:26-31` — the test description says "resets body margin and applies background/foreground/font" but checks for `var(--foreground)` while the actual body rule uses `var(--text-body)`:

```ts
// base.test.ts:26-31
it('resets body margin and applies background/foreground/font', () => {
  expect(base).toContain('body');
  expect(base).toContain('margin: 0');
  expect(base).toContain('var(--background)');
  expect(base).toContain('var(--foreground)');  // FALSE PASS
  // body uses var(--text-body), not var(--foreground)
});
```

The assertion passes because `var(--foreground)` appears in `.vks-wordmark .swarm` (line 124), not in the body rule. The test verifies the wrong thing — it would still pass if `var(--foreground)` were removed from the body rule entirely.

**Evidence**: `remote-frontend/src/styles/tokens/base.test.ts:30`; `remote-frontend/src/styles/tokens/base.css:20` uses `var(--text-body)`.

### F6 [Minor] — `base.test.ts` fragile whitespace assertion

`base.test.ts:37`: `expect(base).toContain('a ')` — checks for literal `a ` (with trailing space). This is fragile: if the selector changes to `a:hover` or `a:focus`, the assertion breaks despite the intent (checking anchor styling) still being satisfied. A regex-based check like `toMatch(/\ba\s*\{/)` would be more robust.

**Evidence**: `remote-frontend/src/styles/tokens/base.test.ts:37`.

### F7 [Minor] — No `vks-pulse` keyframe in remote-frontend

`NodeCard.tsx:48` references `animate-[vks-pulse_2s_ease-in-out_infinite]`. The `@keyframes vks-pulse` definition exists in the design source's `components.css` (which is NOT ported in Phase 1) and in the node `frontend/src/styles/index.css:4-7` (which is a different codebase). The remote-frontend has no `vks-pulse` keyframe definition.

This is pre-existing (the component was already broken), but the design source's `components.css` keyframe uses `hsl(var(--vks-emerald-hsl))` — the HSL triplet approach — which is consistent with the token layer's design. The remote-frontend will need `components.css` (or equivalent) in a later phase.

**Evidence**: `remote-frontend/src/components/swarm/NodeCard.tsx:48`; `dev-docs/designs/2026-07-04-vk-swarm-design-system/design-source/tokens/components.css` contains the keyframe definition.

### F8 [Minor] — Google Fonts `@import` bypasses PWA caching

`fonts.css` does `@import url('https://fonts.googleapis.com/...')`. The PWA's `workbox.globPatterns` (`**/*.{js,css,html,ico,png,svg,woff2}`) doesn't cache external origins. The `runtimeCaching` routes only cover `/v1/`, `/assets/`, and shell routes. Google Fonts CSS and the actual `.woff2` font files are fetched from `fonts.googleapis.com` and `fonts.gstatic.com` respectively — neither is cached by the service worker.

**Failure scenario**: User visits the hive UI, loads fonts, then goes fully offline. On next page load, fonts fall back to system fonts. The design token `--font-ui: 'Inter', 'Noto Emoji', system-ui, sans-serif` has a good fallback chain, so this is a degradation, not a hard break. But the strict "no-external-host posture" from the hive-ui-polish offline-first PWA is implicitly violated.

**Evidence**: `remote-frontend/src/styles/tokens/fonts.css:7`; `remote-frontend/vite.config.ts:15` (globPatterns), `16-44` (runtimeCaching).

---

### F9 [Informational] — No tailwind.config.js color theme mapping

The remote-frontend's `tailwind.config.js` has `theme: { extend: {} }` — no `colors` definition. This means shadcn/ui classes like `bg-background`, `text-foreground`, `border-border`, `bg-[hsl(var(--surface-card))]` rely on:
- `bg-background` → Tailwind looks up `background` as a color name → not found → no CSS generated. CSS variable `--background` is defined but Tailwind doesn't know about it.
- `border-border` → same issue.

This is **pre-existing** (not introduced by this PR). The token layer defines the CSS variables, but Tailwind doesn't generate utility classes that reference them without a theme config mapping. The PR is a step toward fixing this, but it's incomplete without the tailwind.config.js update.

**Evidence**: `remote-frontend/tailwind.config.js:7-9`; `remote-frontend/src/styles/tokens/colors.css:44-45` (variable definitions).
